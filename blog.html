01 왜 만들었는가

개발하다 보면 하나의 AI 답변만으로는 확신이 안 설 때가 있다. ChatGPT에게 물어보고, Claude에게도 물어보고, Gemini에게도 물어보고... 탭을 3개 열어서 같은 질문을 복붙하는 자신을 발견했다.

핵심 동기: "여러 AI의 관점을 동시에 듣고, 서로 반박하게 만들면 더 정제된 결론을 얻을 수 있지 않을까?"

그래서 세운 원칙은 두 가지였다.

  • API 비용 0원 — 이미 구독 중인 Claude Code CLI, Codex CLI, Gemini CLI를 subprocess로 호출. API 키도 필요 없고, 토큰 비용도 없다.
  • Slack 기반 — 이미 매일 쓰는 Slack에서 동작하면 별도 UI를 만들 필요가 없다. 스레드가 자연스러운 토론 구조를 제공한다.

결과적으로 Slack 채널에 메시지 하나 쓰면 3개 AI가 스레드에서 자유 토론을 시작하고, 합의에 도달하면 결론을 채널에 공지해주는 시스템이 만들어졌다.

Socket Mode 선택 이유

일반적인 Slack Bot은 HTTP 엔드포인트를 열어서 Events API를 받는다. 하지만 이 프로젝트는 로컬 PC에서 CLI를 호출해야 하므로 서버가 필요 없는 Socket Mode가 적합하다. WebSocket으로 Slack과 직접 통신하기 때문에 ngrok도 필요 없다.

        tmp.write(prompt)
        tmp.close()
        return tmp.name

합의 감지 알고리즘

각 AI는 응답 마지막에 숨겨진 HTML 코멘트 형태의 합의 JSON을 포함한다.

prompt답변 마지막에 반드시 아래 형식의 합의 JSON을 포함하세요:
<!--CONSENSUS:{"agree": true/false, "summary": "결론 요약 (1~3줄)"}-->

이 JSON을 정규식으로 파싱한다.

pythonCONSENSUS_PATTERN = re.compile(r"<!--CONSENSUS:(.*?)-->", re.DOTALL)

def _parse_consensus(text):
    match = CONSENSUS_PATTERN.search(text)
    if not match:
        return None
    return json.loads(match.group(1).strip())

종료 조건 (3단계)

조건 설명 결과
전원 합의 3개 AI 모두 agree: true 즉시 종료
교착 + 다수 2개 동의 + 최근 2라운드 반복 감지 다수결 종료
최대 라운드 10라운드 도달 강제 종료 + 다수결

교착 상태 감지

최근 2라운드(6개 메시지)의 앞 100자를 비교하여, 3개 중 2개 이상이 동일하면 교착 상태로 판정한다.

python@staticmethod
def _is_stalemate(history):
    ai_msgs = [h for h in history if h["name"] != "사용자"]
    if len(ai_msgs) < 6:
        return False
    recent = set(m["text"][:100] for m in ai_msgs[-3:])
    prev   = set(m["text"][:100] for m in ai_msgs[-6:-3])
    overlap = len(recent & prev)
    return overlap >= 2

라운드 관리

  • 매 라운드마다 발언 순서를 랜덤 셔플하여 특정 AI가 항상 먼저 말하는 편향을 방지
  • "생각 중..." 메시지를 표시했다가 응답 후 삭제하여 UX 개선
  • 라운드 시작 전 스레드에서 사용자 메시지를 수집하여 히스토리에 반영 — 토론 중간에 사용자가 개입 가능
  • 이전 발언은 최근 10개만 프롬프트에 포함하여 컨텍스트 오버플로 방지

04 코딩 모드 상세

#ai-coding 채널에 코딩 요청을 쓰면 3개 AI가 역할을 분담하여 소프트웨어 개발 파이프라인을 실행한다.

Phase 1: 기획 + 설계 + 코드 작성

Claude가 요청을 분석하고, 설계부터 완성된 코드까지 작성한다. 당일 이전 작업 결론이 있으면 컨텍스트에 자동 포함된다.

Phase 2: 코드 리뷰

Codex가 Claude의 코드를 리뷰한다. 버그, 보안 이슈, 개선 사항을 찾아 스레드에 게시한다.

Phase 3: 테스트 작성 (전원 참여)

Codex가 테스트 리더로 전략 수립, Claude가 엣지 케이스 테스트, Gemini가 추가 테스트를 동시에 (asyncio.gather) 작성한다.

Fix Loop (최대 3회)

Codex가 코드+테스트를 분석하여 이슈가 있으면 Claude에게 수정 요청. "이슈 없음"이 나올 때까지 최대 3회 반복한다.

python# modes/coding.py - Fix Loop 핵심
for fix_round in range(1, MAX_FIX_ROUNDS + 1):
    issues_found = await self.codex.ask(
        f"코드와 테스트를 분석하여 이슈가 있는지 판단..."
    )
    if "이슈 없음" in issues_found:
        self._post(channel, thread_ts, "이슈 없음 - 수정 불필요")
        break
    # Claude에게 수정 요청
    current_code = await self.claude.ask(
        f"이슈를 반영하여 코드를 수정해 주세요..."
    )

07 개발 과정에서 겪은 이슈

Issue Windows에서 CLI 호출 방식

처음에는 subprocess.run()으로 직접 호출했지만, Windows에서 긴 프롬프트를 인자로 전달하면 명령줄 길이 제한(8191자)에 걸렸다.

해결: 프롬프트를 임시 파일(.txt)에 저장한 후 type prompt.txt | claude -p 형태로 stdin 파이프로 전달. Windows의 type 명령은 Unix의 cat에 해당한다.

추가: asyncio.create_subprocess_exec()가 아닌 create_subprocess_shell()을 사용해야 파이프(|)가 동작한다. exec은 셸 없이 프로그램을 직접 실행하기 때문이다.

Issue 한글 인코딩

Windows 기본 인코딩은 cp949인데, AI 응답에 유니코드가 포함되면 UnicodeDecodeError가 발생했다.

해결: 환경 변수에 PYTHONIOENCODING=utf-8을 설정하고, stdout.decode("utf-8", errors="replace")로 디코딩 실패 시 대체 문자를 사용.

python# agents/base.py
@staticmethod
def _make_env():
    env = os.environ.copy()
    env["PYTHONIOENCODING"] = "utf-8"
    return env
Issue Slack 이벤트 수신 + 비동기 처리

Slack Bolt의 이벤트 핸들러는 동기 함수인데, AI 호출은 async이다. 핸들러 안에서 await를 쓸 수 없고, asyncio.run()을 쓰면 이벤트 루프 충돌이 발생한다.

해결: threading.Thread로 별도 스레드를 만들고, 그 안에서 asyncio.new_event_loop()로 새 이벤트 루프를 생성하여 코루틴을 실행. 에러가 나면 Slack에 에러 메시지를 전송하는 예외 핸들링도 포함.

pythondef _run_async(coro, client=None, channel=None, thread_ts=None):
    loop = asyncio.new_event_loop()
    asyncio.set_event_loop(loop)
    try:
        loop.run_until_complete(coro)
    except Exception as e:
        # Slack에 에러 알림
        if client and channel and thread_ts:
            client.chat_postMessage(
                channel=channel, thread_ts=thread_ts,
                text=f"오류 발생\n{str(e)[:500]}"
            )
    finally:
        loop.close()
Issue 봇 자신의 메시지 무한 루프

봇이 스레드에 메시지를 쓰면 message 이벤트가 다시 발생하여 자기 자신에게 응답하는 무한 루프가 생긴다. event.get("bot_id")와 event.get("subtype")을 체크하여 필터링.

Issue Gemini CLI의 빈 프롬프트 인자

Gemini CLI는 gemini -p 뒤에 인자가 반드시 필요했다. stdin으로 프롬프트를 전달하더라도 -p "" (빈 문자열)을 붙여야 정상 동작. 이 quirk를 발견하는 데 시간이 걸렸다.

09 Bridge 모드 + Watchdog

v1.2.0에서 추가된 Bridge 모드는 Slack 채널을 특정 프로젝트 폴더의 CLI에 직접 연결한다. #ai-stockradar에 메시지를 보내면 해당 프로젝트 폴더에서 Claude Code가 실행되고, codex: 접두어를 붙이면 Codex CLI로 전환된다.

Watchdog는 봇 프로세스를 15초마다 감시한다. 크래시 시 5초 후 자동 재시작하고, 모든 채널에 알림을 보낸다. !bot status/restart/stop으로 원격 제어도 가능하다. 60초 안에 5번 이상 반복 크래시하면 자동 재시작을 중단하고 수동 개입을 요청한다.

또한 !stop 명령으로 작업을 강제 중단할 수 있다. 스레드에서 실행하면 해당 작업만, 채널 최상위에서 실행하면 해당 채널의 모든 작업을 취소한다.

10 마치며

처음에는 "AI 3개가 싸우면 재밌겠다" 정도로 시작했는데, 실제로 써보니 의외로 실용적이었다. 특히 기술 선택, 아키텍처 설계, 코드 리뷰 같은 주제에서 한쪽으로 치우치지 않는 균형 잡힌 의견을 빠르게 얻을 수 있었다.

전체 코드는 Python 파일 22개, 총 3,100줄 이상이다. 초기에는 500줄 미만이었지만, 병렬 실행, Bridge 모드, Watchdog, 노이즈 필터, 53개 테스트가 추가되며 성장했다. 복잡한 프레임워크 없이 slack-bolt와 asyncio만으로 충분했다.
